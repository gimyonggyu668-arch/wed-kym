1. 현행 시스템 분석 
- 새 시스템을 만들거나 기존 시스템을 개선하기 전, 현재 사용중인 업무 방식, 프로그램, 데이터, 서버, 네트워크, 문제점을 조사하고 정리하는 활동 

분석을 하지 않았을 때 문제점 
- 필요한 기능을 빠트릴 수 있다.
- 데이터 이전 시 오류 발생
- 외부 시스템과 연결 실패
- 새 시스템을 만들어도 만족도가 낮음
- 문제 해결이 아니라 단순 재개발이 된다.

- 회사나 프로젝트에서 정한 규칙을 지켜서 문서를 작성한다.

요구사항 확인 기법

1. 요구사항 검토
- 여러 사람이 모여서 오류, 잘못된 가정, 애매한 표현, 누락 사항, 표준 위반 등을 찾아내는 활동
- 고객쪽 사람이 꼭 1명은 있어야 한다.

2. 프로토 타이핑
- 최종 제품을 만들기 전에 일부 기능이나 화면을 미리 만들어 사용자가 확인하도록 하는 방법

- 종이 프로토타입
- 화면 프로토타입 : 버튼을 누르면 화면 이동정도 되는 프로토타입 
- 기능 프로토타입 :  핵심 기능이나 중요한 기능을 실제와 비슷하게 구현하여 요구사항이 맞는지 검증하는 방식 

3. 모델 검증
- 요구사항 분석 단계에서 만든 각종 모델이 요구사항을 제대로 표현하고 있는지 확인하는 활동

모델 
- 시스템을 이해하기 쉽게 표현한 그림 또는 구조
 
ERD
클래스 다이어그램
시퀀스 다이어그램
화면 흐름도
API 명세

- 요구사항을 글로만 작성하면 사람마다 다르게 이해할 수 있기 때문에 검증이 필요하다.

4. 인수테스트
- 개발이 거의 끝난 후 만든 프로그램이 고객이 요구한 내용을 정말 만족하는지 최종 확인하는 테스트
- 요구사항의 중요한 속성은 제품이 요구사항을 만족시키는 기능인지 확인 가능해야 한다.

요구사항 확인 수행 순서
1. 요구사항 목록에 업무 기능에 대한 요구사항이 모두 반영 되어있는지 확인한다.

- 요구사항 목록 중 수용 여부가 수락인 경우, 요구사항 정의서 (유스 케이스 명세서)가 작성되었는지 확인한다.
- 유스 케이스 명세서가 시스템이 어떻게 동작하여야 하는 지 명확하고 구체적으로 기술하고 있는지 검토해야 한다.

3. 비기능적 요구사항이 명확하게 도출되었는지 확인한다.
- 얼마나 잘 동작해야 하는가를 정하는 요구사항

기능이 아니라 비기능 요구사항으로 잘 적었는지 확인

비기능 요구사항은 정량적이어야 한다. (수치 잘 적어라)

상황별 성능 목표도 구체적이어야 한다.
일반 상황 -> 2초 이내
피크 상황 -> 3초 이내 

타 시스템과 연계 및 인터페이스에 대한 요구사항이 적절히 도출되었는지 확인
- 다른 시스템과 데이터를 주고받아야 한다면 그 연결 조건이 제대로 정리되어 있는지 확인하는 과정

3. 요구사항 확인하기 

복잡성
- 기술이 너무 복잡하거나 우리 시스템 구조와 맞지 않아 문제가 생길 가능성
검증 여부 
- 이 기술이 이미 많이 사용되어 믿고 쓸 수 있는가?
의존성 
- 이 기술이나 업체가 없으면 우리 시스템이 계속 운영되기 어려운가?
특정 업체에 기술에 의존 -> 업체에 문제가 생기면 -> 우리 서비스에도 문제가 생긴다. -> 의존성 위험 높다 

특정 라이브러리만 사용 -> 사용자가 증가해서 라이선스 비용이 오름 -> 예상 비용 증가 

솔루션 
특정 문제를 해결하기 위해 여러 기능과 프로그램을 묶어 만든 제품 또는 시스템
