요구사항 
1단계 : 요구사항 도출 
- 고객이 뭘 원하는지 의견을 듣고 끌어내는 단계 
- 주요기법 : 인터뷰, 설문조사 등
2단계 : 요구사항 분석 
- 도출된 요구사항 중 모호하거나 상층되는 부분을 걸러내고 타당성을 따진다 
- 주요 활동 : 구조적 분석 또는 객체지향 분석(UML, 람보 방법론)을 수행하여 시스템 경계를 정의함 
3단계 : 요구사항 명세 
- 분석된 내용을 바탕으로 개발자와 고객이 모두 승인할 수 있게 문서를 작성하는 단계  
- 주요 활동: 요구사항 명세서(SRS), 유스케이스 명세서 작성. 
4단계 : 요구사항 확인 / 검증 
- 완성된 명세서가 고객이 원하는 게 맞는지, 오류는 없는지  최종 검사 단계 
- 주요 활동: 요구사항 검토회(워크스루, 인스펙션), 인수테스트 계획 수립. 


2. 요구사항 검토(Review) 기법 3가지0
요구사항 명세서(문서)를 검증할 때, 얼마나 엄격하게 회의를 하느냐에 따라 종류가 나뉘어.
동료 검토 (Peer Review): 작성자가 2~3명의 동료들에게 요구사항 명세서를 설명하고 가볍게 결함을 찾는 비공식적인 검토.
워크스루 (Walkthrough): 작성자가 미리 자료를 배포한 뒤, 회의 당일 직접 명세서를 쭉 설명하면서 참가자들과 함께 결함을 찾는 검토회.
인스펙션 (Inspection) (★개중요): 작성자가 아닌 훈련된 중재자(Moderator)가 회의를 진행하며, 미리 작성된 체크리스트에 따라 아주 엄격하게 결함을 찾아내고 공식 기록하는 기법.

3. 요구사항 명세서의 7대 품질 속성
잘 만든 요구사항 명세서가 갖추어야 할 조건들이야. 보기 중 잘못된 성질을 고르는 형태로 출제돼.
명확성 (Clarity): 한 가지 뜻으로만 해석되어야 함.
완전성 (Completeness): 모든 요구사항과 제약조건이 빠짐없이 기록되어야 함.
일관성 (Consistency): 요구사항 간에 서로 모순이나 충돌이 없어야 함.
검증 가능성 (Verifiability): 나중에 진짜 구현되었는지 테스트(수치화)가 가능해야 함.
추적 가능성 (Traceability): 요구사항의 출처와 설계, 소스코드, 테스트 케이스 간의 연결 고리 추적이 가능해야 함.
수정 용의성 (Modifiability): 쉽게 고칠 수 있는 구조여야 함.
서열화 (Ranking): 중요도에 따라 우선순위가 매겨져 있어야 함.

4. 소프트웨어 제품 품질 표준 (ISO/IEC 25010)
비기능 요구사항을 정량적으로 적을 때 기준이 되는 국제 표준 8대 특성이야. (구 ISO 9126의 확장판)
기능적 적합성 (Functional Suitability): 사용자가 요구한 기능을 정확하게 수행하는가.
성능 효율성 (Performance Efficiency): 응답 시간이나 자원(CPU, 메모리) 소모량이 적절한가.
호환성 (Compatibility): 다른 소프트웨어나 하드웨어 환경과 잘 결합하는가.
사용성 (Usability): 사용자가 쉽게 배우고 편리하게 쓸 수 있는가.
신뢰성 (Reliability): 시스템이 장애 없이 얼마나 오랫동안 안정적으로 유지되는가.
보안성 (Security): 불법적인 접근을 막고 데이터 유출을 방지하는가.
유지보수성 (Maintainability): 코드 수정이나 결함 제거가 쉬운가.
이식성 (Portability): 시스템을 다른 운영체제나 기기로 옮기기 쉬운가.
5. 정형 분석 (Formal Analysis) vs 비정형 분석
요구사항을 분석하는 표현 방식의 차이야.
정형 분석: 자연어(한글) 대신 수학적 기호나 논리학 기호를 사용하여 요구사항을 완벽하게 표현하는 방식. 오해가 전혀 없지만 다루기 어려움.
비정형 분석: 우리가 일반적으로 쓰는 자연어(한글)나 다이어그램(그림)으로 표현하는 방식. 이해하기 쉽지만 작업자 간 오해가 생길 수 있음.
💡 마지막 한마디 팁 네가 들고 있던 기본 요약본 내용(현행 시스템 분석, 개발환경 정의, 비기능 수치화 등)에 지금 다시 보여준 이 5가지 뼈대를 얹으면 요구사항 파트는 그냥 마스터한 거나 다름없어.
이 개념들 중에서 특별히 주관식 서술형 예상 예시를 보고 싶거나, 문장으로 길게 쓰는 연습이 필요한 파트가 있으면 말해줘. 바로 뽑아줄게!

- 트렌드 부합성은 현재 개발 환경이나 산업 흐름에 맞는 기술인지 확인하는 것
- 기술을 잘못 선택하면 개발 중 문제가 생길 수 있다.

- 시스템 간 상호 운용성
- 시장 성숙도 및 트렌드 부합성
- 기술적 위험성 분석을을 참조하여 검토 의견을 작성한다.

분석 클래스 모델 타당성 검토

분석 클래스 모델

제어 클래스 : 기능 흐름과 처리 순서를 담당하는 클래스 
엔티티 클래스 : 실제 업무 데이터를 저장하고 관리하는 클래스 


타당성 분석 결과를 이해관계자가 검증한다.
1. 타당성 분석 결과를 관련 이해관계자에게 배포하여 사전 검토를 요청한다. 
2. 관련 이해관계자가 모여 분석모델 타당성 분석 결과를 검증한다.
3. 타당성 분석 결과에 이견이 있는 경우 프로젝트 관리자의 중재하에 합의를 도출한다. 

관련 이해관계자 검증을 거친 타당성 분석 결과를 확인하고 배포 및 공유한다. 
1. 관련 이해관계자 검증을 거친 타당성 분석 결과를 의사 결정자가 확인한다.
2. 의사 결정자가 확인한 타당성 분석 결과를 이해관계자에게 배포하여 공유한다.
