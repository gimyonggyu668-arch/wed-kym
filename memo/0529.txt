# 실무 시나리오

A 모바일 주문 서비스는 주문 요약 정보를 여러 프로그램에서 공통으로 사용하고 있다.

현재 개발팀은 다음 문제를 해결하려고 한다.

- 주문 조회 SQL이 여러 화면에 반복 작성되고 있다.
- 일반 직원은 회원 이메일(email)을 조회할 수 없어야 한다.
- 조회 결과를 하나의 가상 객체 형태로 재사용하려고 한다.
- 기존 객체를 삭제하지 않고 수정 가능한 구조로 운영하려고 한다.
- 해당 객체를 통해 데이터 수정은 발생하지 않아야 한다.

현재 시스템에는 다음 테이블이 존재한다.

## members 테이블

| 컬럼명 | 데이터 타입 |
| --- | --- |
| member_id | INT |
| member_name | VARCHAR(50) |
| email | VARCHAR(100) |

## orders 테이블

| 컬럼명 | 데이터 타입 |
| --- | --- |
| order_id | INT |
| member_id | INT |
| order_date | DATE |

회원과 주문은 회원번호 기준으로 연결되어 있다.

# 문제

위 요구사항을 만족하도록 주문 요약 정보를 제공하는 View를 생성하시오.

# 조건

- DBMS는 MySQL 기준으로 작성한다.
- View 이름은 `v_order_summary` 로 작성한다.
- 조회 컬럼은 다음만 포함한다.

컬럼명

---

order_id

---

member_name

- 회원과 주문 데이터를 함께 조회해야 한다.
- SQL은 실제 실행 가능한 형태로 작성한다.

물리 데이터저장소 구성
관계형 데이터베이스 사이에서의 제약조건
PK - FK 연결되어 있는 두 개 테이블의 데이터에 대한 제약조건

삭제 했을 때 (ON DELETE)
CASCADE : PK 값이 삭제되면 FK 쪽에서도 같이 삭제된다.
RESTRIC : FK쪽에서 사용중이면 PK쪽의 값을 삭제할 수 없다.
NULLIFY : PK쪽 값이 삭제되면 FK쪽에서는 NULL로 바꾼다.
수정했을 때 (ON UPDATE)
CASCADE : PK쪽 값이 수정되면 FK쪽에서도 같이 수정된다.
RESTRICT : FK쪽에서 사용중이면 PK쪽의 값을 수정할 수 없다.
NULLIFY : PK쪽 값을 수정하면 FK쪽에서는 NULL로 바꾼다.

언제쓰냐? 
테이블 만들 때 외래키 제약조건 밑에 같이 속성을 작성해준다.

인덱스 설계 
- 조회 성능을 높이기 위해 어떤 컬럼에 인덱스를 적용시켜줘야 할까?

인덱스 컬럼의 분포도가 10 ~ 15% 이내가 일반적이다.
분포도의 값이 적을수록 인덱스 효율이 높다 -> 분포도가 넓다.

조회 및 출력 조건으로 사용되는 컬럼인 경우 
- WHERE 에서 많이 사용되는 컬럼인가 

수정이 빈번하지 않은 컬럼을 선정을 한다.

자주 조합되서 사용되는 컬럼은 결합 인덱스로 생성 



뷰(View)
- SELECT를 한 결과를 저장하는 가상의 테이블 

REPLACE
- 뷰가 이미 존재하는 경우 기존 뷰를 새 내용으로 다시 생성하는 속성


FORCE
- 기존 테이블이 존재하지 않아도 뷰를 생성하는 속성
- ORACLE에서 사용됨
- 단, 뷰를 사용하려고 하면 에러가 발생 

NOFORCE
- 기본테이블이 존재할때만 뷰를 생성할 수 있게 함 

클러스터 
- 자주 같이 조회되는 데이터를 디스크의 가까운 위치에 저장해서 조회 속도를 높이는 물리적 저장 방법 

분포도가 넓을 때 유용하다

여러 블록 이상의 테이블에 적용한다.
블록 : DB가 데이터를 저장할 때 블록이라는 저장 공간 단위로 저장한다.

- 데이터가 많은 경우
디스크 내에서 수천 ~ 수만개 블록으로 나뉜다. 


관련 데이터를 가까운 블록에 모아둠 

디스크 이동이 감소 -> 읽는 블록 수 감소 -> 조회 속도가 높아진다. 
  




파티션 설계
- 하나의 테이블 내에서 내부적으로 여러 조각으로 저장하는 기술
- 하나의 테이블만 사용되지만 DBMS는 데이터를 여러 저장구역으로 나누어 관리한다.

종류 
범위 분할

해쉬 분할

조합 분할


디스크 구성 설계 
- 데이터를 어디에, 어떻게 저장할 것인지 계획하는 작업

많은 작업이 발생하는 데이터를 한 곳에 몰아두지 않는다. 

입출력 경합을 최소화 하여 데이터의 접근 성을 올린다.
- 입출력 경합 : 여러 작업이 같은 디스크를 동시에 사용하는 상황
