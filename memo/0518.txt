운영체제  : 운영체제는 하드웨어와 사용자 간의 사이의 연결을 도와주는 핵심적인 시스템 소프트웨어다.
운영체제 종류 
- windows
- unix
- linux
- ios

1. 가용성
- 장기간 시스템을 운영할 때 고장이 안 나는가 
- 특정 프로그램의 메모리 누수로 인한 성능 저하가 없는가 
- 운영체제 보안상 허점으로 반복적인 패치 설치를 위한 재기동
- 운영체제의 버그 등으로 인한 패치 설치를 위한 재기동 
2. 성능 
- 대규모 동시 사용자 처리를 안정적으로 할 수 있는가 
- 대규모 파일 작업 처리가 가능한가 
- 지원 가능한 메모리 크기 (32bit, 64bit)
3. 기술 지원
- 공급 벤더들의 안정적인 기술지원(문제가 생겼을 때 도움을 잘 주는가)

4. 상호 호환성
- 설치 가능한 하드웨어 
- 다수의 주변기기 지원 여부 (프린터, 스캐너, usb, 그래픽카드)
5. 구축비용
- 지원 가능한 하드웨어 비용(cpu 요구사항, 메모리 요구사항, 저장장치 요구사항)

1. 요구사항 검토
- 여러 사람이 모여 요구사항 문서를 읽고 문제를 찾는 다.
오타, 잘못된 가정, 애매한 표현, 누락, 모순, 표준위반점을 찾는다. 
- 감토자 그룹을 어떻게 구성하느냐가 중요하다 
예를 들어 고객 중심 프로젝트에서는 검토자 그룹에 고객 대표자가 1명 이상 포함 되어야 한다. 
- 검토는 시스템 정의서, 시스템 사양서, 소프트웨어 요구사항 명세서를 완성한 시점에서 이루어진다.

시스템 사양서

 2. 프로토타이핑(Prototyping)
- 최종 제품을 만들기 전에 일부 기능이나 화면을 미리 만들어 사용자가 확인하도록 하는 방법

장점
- 분석가의 가정을 파악하고 잘못된 경우 유용한 피드백을 제공한다.
- 사용자 인터페이스의 동적인 행위가 문서나 그래픽 모델 보다 프로토타입으로 이해하기 쉽다. 
- 요구사항의 가변성이 프로토타이핑 이후에 급격히 감소한다.
- 잘못된 요구사항에 대해 자원 낭비 방지 

단점
- 핵심 기능보다 디자인에 집중하게 될 수 있다. 
- 프로토타입을 만드는 시간과 비용이 추가로 발생할 수 있다.

모델 검증
- 개발 전에 만든 분석 모델이 올바르게 작성 되었는지 확인하는 방법
- 개발 전에 모델 오류를 찾기 위해서 검증을 해야된다.
- 예를 들어 객체 모델의 경우 객체들 사이에 존재하는 의사소통 경를 검증하기 위해 정적 분석을 수행하는 것이 유용하다.

모델 : 실제 프로그램 코드가 아니라 개발 전에 만드는 분석용 설계 자료

모델의 종류 
업무 흐름도 : 어떤 순서로 업무를 처리하는지 나타내는 그림
유스케이스 다이어그램 : 사용자가 시스템으로 무엇을 할 수 있는지 나타낸 그림 
객체 모델 : 시스템 안에 어떤 객체들이 있고 서로 어떻게 연결되는지 나타낸 모델
클래스 다이어그램 : 클래스, 속성, 메서드, 관계를 표현한 그림
데이터 모델 : 데이터베이스에 어떤 데이터가 저장되는지 나타낸 모델


의사소통 경로
- 객체들끼리 정보를 주고 받을 수 있는 연결 관계
예 : 도서 대여 시스템에서 회원이 도서를 빌려면 다음과 같이 동작해야 한다.
회원 -> 대여 -> 도서

회원이 대여를 요청하고, 대여 객체는 도서 객체에게 재고나 대여 가능 여부를 확인해야 한다.
이때 대여와 도서 사이에 연결이 없다면 문제가 생긴다.
회원 -> 대여      도서 
이렇게 되어 있으면 대여 객체가 도서 정보를 확인할 길이 없다.
필요한 의사소통 경로가 빠진 상태라고 한다.

정적분석
- 프로그램을 직접 실행하는 것이 아닌, 구조만 보고 검사하는 방식
- 객체 모델을 실행할 수는 없지만, 그림이나 관계를 보고 구조적으로 검사하는 것
