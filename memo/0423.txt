통신은 요청  < - >  응답 
요청(request)을 하는 쪽을 클라이언트 (Client)라고 한다.

응답 (reponse) 을 하는 쪽을 서버(Server)라고 한다.

http는 한 번 연결을 하고 나면 기억을 하지 않는다.
1. 클라이언트가 요청을 보냄
2. 서버가 응답을 보냄
3. 연결 종료(끝)

서버 입장에서는 " 이 사용자가 방금 전에 뭘 했는지를 모른다."
ex) 
로그인 요청 -> 성공 
다음 페이지를 달라고 요청하는데 " 너 누군데? "(다시 인증을 요구)

로그인이 유지가 안됨
장바구니가 유지가 안됨
사용자 구분 불가능

상태를 기억하기 위해 별도의 기술을 활용해야 한다. 

1. 쿠기
- 브라우저에 저장되는 데이터 
서버 -> 쿠키를 발급
브라우저 -> 요청할 때마다 쿠키를 같이 자동으로 전송 

2. 세션(session)
서버가 사용자 상태를 기억
java에 Session 클래스가 있다.
세션 객체를 만들어서 사용자 상태를 저장한다.

3. 토큰 (jwt)
- 요즘 가장 많이 사용하는 방식 
로그인 성공 -> 토큰 발급
클라이언트가 저장 
요청시 토큰을 같이 보냄 

사용자가 웹에  www.example.com

@GetMapping("경로")

MSA(Mivroservice Archietcture)
- 분산 시스템을 구성하기 위해 하나의 큰 어플리케이션을 여러 개의 작은 서비스로 나눠서 개발하고 운영하는 구조
- 각각의 서비스는 독립적으로 실행된다.
- 서비스 끼리는 네트워크로 통신(HTTP 등)
- 배포도 각각 따로 가능함

분산 시스템 SW

시스템 
- 여러 요소가 모여 하나의 기능을 수행하는 구조
분산
- 한 곳에 몰아두지 않고 여러 곳에 나누는 것
SW
- 그 구조가 실제로 동작하도록 만드는 소프트웨어

여러 컴퓨터에 기능을 나누어 배치하고, 이들이 서로 통신하면서 하나의 시스템  처럼 동작하게 하는 소프트웨어

왜 분산 시스템 SW가 필요할까
- 한 대의 서버로도 프로그램은 만들 수 있다.
사용자가 많아지고, 기능이 많아지고, 장애에 강해야 하면 한 대만으로는 한계가 생긴다. 

쇼핑몰에 사용자가 갑자기 10만명이 몰리면, 한 서버가 감당하기 어렵다.
서버1 : 로그인
서버2: 상품 목록
서버3: 결제 
서버4: 리뷰



한 대의 서버가 있으면 그 서버가 죽는 순간 서비스 전체가 멈춘다.
여러 서버로 나누면 일부 서버가 고장나도 서비스가 완전히 멈추지 않도록 설계할 수 있다.

분산 시스템 SW 구성요소 

1. 어플리케이션 서버
- 실제 비즈니스 로직을 처리하는 서버 
EX) 스프링부트 , Django

비즈니스 로직 : 백엔드 기능 (규칙/판단/계산)

2. 데이터베이스 서버 
- 데이터를 저장하는 서버
MYSQL
ORACLE

분산환경에서는 데이터베이스도 하나만 쓰지 않고 나누거나 복제하기도 한다.

3. 로드 밸런서 
들어오는 요청을 여러 서버에 나누어주는 장치 또는 소프트웨어 
NginX

단점 
- 구조가 복잡해진다.
- 네트워크를 타다보니, 지연, 끊김, 타임아웃 문제가 생길 수 있다.
- 데이터베이스를 나누다보면 데이터가 안맞을 수 있다.

EAI / ESB
서로 다른 성격의 시스템을 연결해주는 방식

회사에서는 프로그램을 하나만 쓰지 않는다.
홈페이지는 주문을 받고
창고 시스템은 재고를 관리하고 
회계 시스템은 매출을 계산하고
배송 시스템은 배송 상태를 관리한다.

이 시스템들이 서로 연결되지 않으면 사람이 직접 옮겨 적거나, 파일을 수 작업으로 옮겨야 한다.


EAI (Enterprise Application Integration)
회사 안의 여러 프로그램을 서로 연결해서 하나처럼 동작하게 만드는 방식

구조
허브 앤 스포크 (Hub and Spoke)
Hub는  중심
Spoke : 자전거 바퀴 살 처럼 바깥으로 연결된 구조 

ESB(Enterprise Service Bus)
EAI가 중앙 허브 중심이라면 ESB는 공통 통신 통로(버스)를 만들어서 각 시스템이 버스를 이용해 유연하게 통신하는 방식 

특징 
표준 기반 통신
서로 다른 시스템들이 같은방식으로 통신할 수 있게 한다.

느슨한 결합
시스템끼리 아주 강하게 묶이지 않게 만든다.

메시지 중심 구조
데이터를 메시지 형태로 주고 받는다.

라우팅 기능 
어떤 메시지를 어떤 시스템으로 보낼지 결정

변환 기능
데이터 형식을 바꿔준다.

확장성이 좋다
 서비스가 늘어나도 비교적 유연하게 붙일 수 있다.
