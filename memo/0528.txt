많은 인덱스는 오버헤드가 된다.
오버헤드 : 추가 부담
- 인덱스가 많으면 조회는 빨라질 수 있지만, 변경작업은 느리다.

예를 들어 회원 1명을 추가할 때 테이블에만 저장하는 것이 아니라 여러 인덱스도 함께 갱신해야한다.
ex) 인덱스가 5개라면, 회원 데이터 1건을 넣을 때 인덱스 5개도 함께 관리해야 한다.

인덱스는 추가 저장공간이 필요하다
- 테이블과 별도로 저장되는 검색 구조이다.
- 인덱스를 만들면 저장 공간을 추가로 사용한다.
- 그래서 무조건 많이 만드는 것은 좋지 않다.

넓은 범위를 인덱스로 처리하면 오히려 느릴 수 있다.
- 인덱스는 적은 데이터를 빠르게 찾을 때 효과적이다.
- 하지만 전체 데이터의 많은 부분을 조회하면 오히려 전체 테이블을 읽는 것이 더 빠를 수 있다.

뷰(View)
- 실제 데이터를 저장하는  테이블이 아니라, select문을 저장해 둔 가상의 테이블
- 자주 사용하는 조회 sql에 이름을 붙여 둔 것

뷰가 필요한 이유
- 복잡한 SQL을 단순하게 사용할 수 있다.
ㄴ 여러 테이블을 JOIN하는 SQL은 길고 복잡하다.

- 특정 컬럼만 보여줄 수 있다.
ㄴ 개인정보, 급여와 같은 민감한 정보는 제외하여 뷰를 만들면 필요한 컬럼만 공개할 수 있다.

뷰 속성 
replace
 - 뷰가 이미 존재하는 경우 기존 뷰를 새 내용으로 다시 생성하는 속성 

force
- 기본 테이블이 존재하지 않아도 뷰를 생성하는 속성 

create force view member_view as
select member_id, name from member;


이 경우 Oracle에서는 테이블이 없어도 View 정의 자체는 만들 수 있다.
다만 실제로 조회하려고 하면 오류가 발생할 수 있다.



NOFORCE
- 기본 테이블이 존재할 때만 뷰를 생성하는 속성
CREATE NOFORCE VIEW member_view as
select member_id, name from member;
- Oracle에서 기본값은 보통 NOFORCE이다.
- 일반적으로는 참조할 테이블이 있어야 뷰를 만들 수 있다.

WITH CHECK OPTION 
- 뷰의 조건을 만족하는 행만 INSERT 또는 UPDATE 할 수 있도록 제한하는 속성

- 활성 회원만 저장하는 VIEW
CREATE VIEW active_member_view as
select member_id, name,  status
from member
where status = 'active'
WITH CHECK OPTION


상태를 건드리는  수정은 할 수 없다.

WITH READ ONLY 
- 뷰를 조회 전용으로 만드는 속성
CREATE VIEW active_member_view as
select member_id, name, status
from member
where status = 'active'
WITH READ ONLY;


뷰 설계시 고려사항
뷰는 결국 테이블을 조회한다.
- 실제 데이터를 저장하지 않는다.
- 실제로는 내부에서 원본 테이블의 SELECT가 실행되는 것과 같다.
- 그래서 뷰를 사용한다고 해서 무조건 빨라지는 것은 아니다.
- 오히려 뷰 안의 SQL이 복잡하면 성능 문제가 생길 수 있다.

복잡한 뷰는 성능이 느려질 수 있다.
- member, orders 테이블을 조인하는 쿼리문을 view로 만든다.
- 뷰를 조회하게 되면 실제로는 member와 orders테이블을 조인하고 집계한다.
- 데이터가 많으면 이 작업은 느려질 수 있다. 

뷰 안의 select 조건도 인덱스를 고려해야 한다.
- 뷰 안에 조건이 있다면, 해당 조건 컬럼에 인덱스가 있는지 확인해야 한다.
CREATE VIEW active_member_view as
select member_id, name, email
from member
where status = 'active';
- 이 뷰는 status 조건을 사용한다.
- 따라서 status 컬럼이 자주 조회되고 분포도가 적절하다면 인덱스를 고려할 수 있다.
- 다만 status 같이  ACTIVE와 INACTIVE 처럼 종류가 적고 대부분이 ACTIVE라면 인덱스 효과가 낮을 수 있다. 

뷰를 사용하는 SQL의 WHERE 절도 중요하다
- 뷰 안의 조건이 부족하더라도, 뷰를 조회하는 SQL에서 좋은 조건을 줄 수 있다. 
CREATE VIEW order_view as
SELECT order_id, member_id, order_date, total_price
from orders;
- 모든 주문을 보여주는 뷰 
- 그런데 실제 조회할 때는 다음 처럼 조건을 준다.
SELECT*
FROM order_ciewe
WHERE member_id = 10;
- 이 경우 실제로는 orders.member_id를 조건으로 조회하게 된다.
- 따라서 orders.member_id에 인덱스가 있으면 성능이 좋아질 수 있다.

인덱스 설계

인덱스 적용 기준
컬럼의 분포도가 10~15% 이내인 경우
분포도 : 해당 컬럼의 값이 얼마나 다양하게 나뉘어 있는지 의미 
분포도 = (1/컬럼의 도메인 개수) x 100

회원번호 -> 모두가 다르다
10000

분보포 = (1/10000) x 100 = 0.01 %

- 이런 컬럼은 인덱스 효과가 크다

10% 이하 : 점점 효율이 좋아진다.
10~15% : 일반적으로 사용할 수 있다.
15% 이상으로 올라갈수록 : 인덱스 효율이 안좋아진다.

인덱스 컬럼의 선정
- 분포도가 좋은 컬럼은 단독 인덱스로 생성한다.
분포도가 좋다는 것은 검색 결과가 적게 나오는 컬럼이라는 뜻

수정이 빈번하지 않은 컬럼을 선정한다.
- 인덱스는 조회를 빠르게 해주지만, 삽입, 수정, 삭제 작업에는 부담이된다.
- 인덱스로 선정한 컬럼의 정리 내용을 따로 저장을 하기 때문에 테이블 데이터가 바뀔 때 인덱스도 같이 바뀌어야 한다. 

설계시 고려사항
새 인덱스가 기존 액세스 경로에 영향을 줄 수 있다.
- 액세스 경로 : DBMS가 데이터를 찾기 위해 선택한 이동 방법
대표적인 액세스 경로
1. Full Table Scan(전체 테이블 검색)
- DB가 처음부터 끝까지 전부 읽는다.
2. Index Scan(인덱스 검색)
- 인덱스를 사용해서 필요한 위치만 바로 찾는다.

인덱스는 추가 저장공간이 필요하다.
- 인덱스는 테이블과 별도로 저장되는 검색 구조이다.
- 인덱스를 만들면 저장 공간이 추가로 사용한다.
 
넓은 범위를 인덱스로 처리하면 오히려 느릴 수 있다.
- 적은 데이터를 빠르게 찾을 때 효과적이다.
- 하지만 전체 데이터의 많은 부분을 조회하면 오히려 전체 테이블을 읽는 것이 더 빠를 수 있다.

인덱스와 테이블 데이터의 저장 공간을 적절히 분리한다.
- 대규모 DB에서는 테이블 데이터와 인덱스 데이터를 물리적으로 분리해서 저장하기도 한다.

SQL 작성 방법

파티션을 만들어 놓고 평소대로 사용하듯이 해도 DB가 알아서 조회 

2024~2026년의 데이터를 가지고 있을 때
각 년도별로 파티션을 분할했다면
2025년 데이터만 조회 
-> 2025년 데이터가 들어있는 파티션만 확인 
-> 2024, 2026년 데이터가 들어있는 파티션은 확인하지 않는다.

데이터를 추가할 때 특정 파티션에 추가하는게 아니라 DB가 자동으로 계산해서 저장한다.

해시 분할의 장점
데이터 분산 : 특정 파티션에 데이터가 몰리는 현상을 감소시킨다.
저장 균형 : 여러 파티션 크기를 비슷하게 유지
특정 키 조회 빠름 : ID 기본 조회에 적합
대용량 처리 : 부하를 여러 파티션으로 분산할 수 있다. 

해시 분할의 단점 
범위 조회에 약함 : 날짜 범위 검색 비효율
사람이 예측 어렵다 : 어느 파티션에 있는지 알기 어렵다.
파티션 개수 변경이 어렵다 : 재분배 작업이 발생할 수 있다.

조합 분할 (Composite Partitioning)
- 파티션 기준을 2개 이상 조합해서 데이터를 나누는 방식
- 먼저 큰 기준으로 한 번 나누고, 그 안에서 다시 작으 ㄴ기준으로 한 번 더 나누는 방식 

조합 분할을 사용하는 이유
- 대용량 데이터 관리 : 1차 파티션만으로 부족할 때 사용
- 조회 범위 축소 : 날짜 조건과 고객 조건을 함께 자주 조회할 때 좋다
- 데이터 분산 : 특정 연도나 월에 데이터가 올리는 문제를 줄일 수 있다.
- 관리 편의성 : 연도별 삭제 , 백업 같은 작업을 하면서 내부 분산도 가능하다.

언제 사용하면 좋을까?
- 주문 테이블 1차 : 주문일 2차 : 고객번호
- 로그 테이블 1차 : 생성일 2차 : 사용자 ID
- 결제 테이블 1차 : 결제일 2차 : 회원 ID
- 게시글 이력 테이블 1차 : 작성일 2차 : 작성자 ID

파티션 장점
- 읽어야 하는 데이터 범위를 줄여서 성능을 향상
- 전체 데이터 훼손 가능성 감소 및 데이터 가용성 향상
- 각 분할 영역을 독립적으로 백업하고 복구가 가능하다.
- 디스크 스트리핑으로 I/O성능 향상
디스크 스트리핑 : 데이터를 여러 디스스켕 나눠서 동시에 저장하거나 읽는 방식


파티셔닝 순서
- 파티션 종류 결정 (범위, 분할,조합)
ㄴ 데이터를 어떤 방식으로 나눌지 결정
범위 분할 : 값의 범위로 나눈다
해시 분할 : 해시 계산 결과로 나눈다
조합 분할 : 두 가지 방식을 함께 사용
- 파티션을 할 키를 설정
ㄴ 데이터를 나눌 기준이 되는 컬럼을 결정
- 파티션의 개수를 결정
데이터의 양 : 한 파티션이 너무 커지지 않게 한다
조회 패턴 : 자주 조회하는 단위와 맞춘다.
보관 정책 : 삭제, 백업 단위와 맞춤
관리 편의성 : 너무 많은 파티션은 피한다.
I/O분산 : 특정 파티션에 너무 몰리지 않게 한다.

클러스터 설계 
- 자주 조회되는 데이터를 디스크의 가까운 위치 모아 저장해서 조회 속도를 높이는 물리적 저장방법이다.

일반 저장 방식은 책이 아무 책장이나 흩어져 있는 상태 클러스터 저장 방식은 자주 같이 보는 책들은 같은 책작이나 가까운칸에 모아두는 방식으로 한다
                                                                                                                                                                                                                                                                                                      파티션 : 저장공간을 나눈다.
클러스터 : 관련 데이터를 가까이 저장한다.


클러스터 설계 방법
 Oracle 기준  

CREATE CLUSTER  CUSTOMER_ORDER_CLUSTER(
CUSTOMER_ID NUMBER
)
CUSTOMER_ID가 같은 데이터 끼리 공간에 저장할 창고를 만들어라 



create table orders(
order_id,
customer_id
)
cluster customer_order_cluster(customer_id);

같은 customer_id를 가진 데이터가 물리적으로 가까운 위치에 저장될 수 있다.


MySQL
- 오라클 처럼 CREATE CLUSTER를 통해 클러스터를 생성하지는 않는다.
- InnoDB 테이블의 PRIMARY KEY가 자동으로 클러스터형 인덱스가 된다.
- MySQL 공식 문서에서도 InnoDB는 테이블에 prinary key를 정의하면 그것을 clustered
index로 사용한다고 적혀있다.

InnoDB
- MySQL에서 테이블 데이터를 저장하고 관리하는 저장 엔진이다.

PK를 만들면 곧 클러스터가 된다.

적용 기준
분포도가 넓을수록 유리한 기법
- 일반적으로 인덱스는 조건으로 조회할 때 데이터가 적을수록 유리하다.
- 그런데 조회범위가 너무 넓으면 인덱스를 타도 많은 행을 계속 찾아가야 해서 오히려 부담이 된다.

select*
from orders
where order_date between '2026-01-01' and '2026-03-31'

- 3개월치 주문 데이터가 매우 많다면 인덱스로 하나하나 찾아가는 비용이 커진다.
- 이런 경우 관련 데이터를 물리적으로 가까이 모아두는 클러스터가 도움이 된다. 


클러스터는 액세스 기법이 아니라 물리적 저장방법이다.
select *
from orders
where customer_id = 10;
- 클러스터는 SQL문법에서 직업 "클러스터로 검색해라"라고 지시하는 개념이 아니다.
- 클러스터는 미리 데이터를 저장할 때 관련있는 데이터를 가깝게 저장한다. 

customer_id = 1인 주문들
customer_id = 2인 주문들
customer_id = 3인 주문들
- 이렇게 같은 키 값을 가진 데이터들이 모이면, 반복되는 키 관리나 블록 접근 측면에서 저장 효율이 좋아질 수 있다.
- 단, 무조건 저장 공간이 줄어드는 것은 아니다.
- 설계가 잘못되면 오히려 공간 낭비가 생길 수 있다.

작은 테이블 
-> 그냥 전체 조회해도 빠름
-> 클러스터 설계 효과가 적다


큰 테이블
-> 여러 블록에 데이터가 흩어질 수 있다.
-> 클러스터 설계효과가 커질 수 있다.

- 클러스터는 데이터가 충분히 많고, 조회 패턴이 반복적으로 발생할 때 의미가 있다. 

인덱스를 사용한 처리가 부담되는 넓은 분포도에 활용한다
- 조회할 데이터가 너무 많으면 인덱스에 반복접근이 많아져서 부담된다.
- 클러스터는 관련 데이터를 물리적으로 모아두므로, 넓은 범위를 읽을 때 블록 접근 수를 줄이는데 도움이 된다.

여러 테이블이 빈번히 조인될 때 사용

정규화 때문에 분할된 반복 컬럼에 활용된다.

설계 고려사항 
- 검색은 빨라지지만 입력, 수정, 삭제는 느려질 수 있다.
- 새로운 데이터를 넣을 때 어디에 넣어야 같은 그룹끼리 모일까를 고려해야 한다.
- DB는 단순히 빈 공간에 넣는 것이 아니라 , 클러스터 구조를 유지해야 한다.  

union, distinct, order by, group by 가 빈번한 컬럼으로 고려한다.
- 이 연산들은 데이터를 묶거나 정렬하거나 중복제거 하는 컬럼이다.

select * 
from orders
oder by order_date;


order_date 기준으로 데이터가 어느정도 모여있으면 정렬 부담이 줄어들 수 있다. 


하지만 무조건 적용되는 것은 아니다.
해당 컬럼으로 조회, 정렬, 그룹화가 자주 발생하는지 확인해야 한다. 

수정이 자주 발생하지 않는 컬럼이 좋다.
- 클러스터 기준 컬럼은 자주 바뀌지 않는 것이 좋다.
- 회원 id -> 거의 바뀔일 없음 
반면 status는 자주 바뀔 수 있다.
결제 대기 -> 결제 완료 -> 배송중 -> 배송완료
