UI 설계 내용 확인
- 개발자가 화면을 만들기 전 "이 설계서 대로 만들면 되는지"를 점검하는 단계
UI 상세 설계 
- 요구사항과 표준 지침에 따라, 메뉴 구조를 설계하고 시스템 단위의 내외부 화면과 폼을 설계할 수 있다.

UI 시나리오 작성 원칙
- UI 상세 설계에 있어서 시나리오 작성은 반드시 필요한 사항이다.
- 화면 구성뿐만 아니라, 사용 흐름, 상호작용, 오류 처리, 접근성, 적용 수준까지 함께 설계 해야한다.

1. 사용자 상호작용 중심 설계 
- 시나리오의 출발점은 항상 "사용자 행동"이어야 한다.
- 핵심 기능과 서비스 별로 사용자의 상호작용 흐름을 정의하고 주요 단계를 시나리오 형태로 설계한다.

2. 적용 수준 구분
- 서비스에 적용할 기능이나 UI 요소를 필수, 권장, 우수로 나누어 정리하는 것 
- 한 번에 다 개발할 수 없기 때문에 반드시 구현해야 하는 기능, 추후 개선할 기능을 구분해야 한다.
- 필수 : 서비스 제공을 위해 반드시 필요한 수준
- 권장 : 사용사 편의성을 높이기 위해 적용하는 수준
- 우수 : 더 나은 사용자 경험을 제공하기 위한 고도화 수준 

3. 접근성과 포용성 반영
- 각 시나리오 단계에는 접근성 기준을 적용하며, 고령자와 장애인을 포함한 다양한 사용자 환경을 고려하여 설계한다.
라벨 제공 : 어떤 정보를 입력해야 하는지 명확히 표시
키보드 제어 : 마우스 없이 Tab, Enter키로 조작할 수 있게 하는 것
색상 대비 : 배경과 글자를 잘 구분할 수 있게 하는 것 
오류 안내 : 오류 원인과 해결 방법을 함께 제공한다.
대체 텍스트  : 이미지나 아이콘에 설명을 같이 제공을 하는 것

4. 상호작용, 피드백, 오류 처리
피드백 : 사용자가 화면에서 행동을 했을 때, 그에 대한 반응
- 사용자의 행동 이후 시스템의 반응, 입력 오류 처리, 장애 발생 시 대응 절차를 시나리오에 포함한다. 

피드백 예시 
버튼 클릭 : 메시지 표시
필수 항목 누락 : 누락된 항목 위치로 이동시키고 오류 메시지를 표시 
서버 오류 : "오류가 발생했습니다, 잠시 후 다시 시도해주세요" 안내

오류 처리의 예시 
필수 항목에 대한 미입력 -> xxx를 입력해주세요 -> 해당 입력창에 포커스 이동
로그인 실패 -> 아이디나 비밀번호가 올바르지 않습니다. -> 로그인 화면을 유지  

5. 공통 요소와 반복 패턴 활용 
- 시나리오는 가이드라인에서 제시한 컴포넌트, 기본 패턴, 스타일 가이드를 기반으로 작성하고 재사용이 가능하도록 구조화한다.

컴포넌트 : UI를 구성하는 재사용 가능한 부품 (버튼, 입력창, 메뉴)
기본패턴 : 사용자가 자주 경험하는 화면 흐름 
스타일 가이드 : 버튼, 색상, 폰트, 용어, 글자크기 등을 어떻게 사용할지 정한 규칙 

6. 실제 사용자 데이터 기반 구성
- 설문, 사용자 의견, 행동 데이터, 평가 결과 등을 근거로 실증적인 사용 패턴과 문제 상황을 반영하여 시나리오를 작성한다.

7. 화면 사용 도움 정보 포함 
- 화면 내 도움말 정보는 흐름도와 시나리오에 포함하여, 특히 복잡한 흐름은 사용자가 명확한 안내를 받을 수 있도록 설계

UI 시나리오 문서의 작성 요건
- UI 시나리오 반드시 포함해야 하는 내용
1. UI 시나리오 문서 목적 및 대상 사용자 
목적 : 핵심 사용자의 흐름과 과업 중심 상호작용을 명확하게 기술하여, 실제 서비스 구현 시 혼선이 발생하지 않도록 한다. 

대상 사용자 : 기획자, 디자이너, 퍼블리셔, 개발자 등 화면 설계에 참여하는 모든 인원


2. 구성 요소
- 개요 : 사용자 목표, 시나리오 목적, 기본 전제조건 제시
- 사용 흐름 : 단계별 화면 이동, 상태 변화, 화면 전환 로직 
- UI 요소 구성 : 아이디어 도출 및 설계 방향성 탐색
- 상호작용 및 입력 로직 : 버튼 클릭, 입력 필드, 오류 상황 대응, 피드백 흐름 정의
- 접근성 고려 사항 : 웹 접근성 기준 준수, 키보드 제어, 명확한 라벨과 판독성 확보
- 예외/오류 시나리오 : 오류 발생 시 흐름, 복구 절차, 사용자 안내 메시지 설계 
- 적용 수준 : 필수, 권장, 우수 수준으로 구분하여 단계 적용 계획 수립 

3. 작성 원칙
- 일관된 표현 사용 : 디자인 원칙과 스타일 가이드라인에 따른 명칭과 용어를 일관되게 사용한다.
- 사용자 중심 : 화면 흐름은 실제 사용자가 수행하는 이벤트 단계를 중심으로 기술한다.
- 현장 적용 가능 : UI 요소의 위치, 텍스트 라벨, 상호 작용 과정의 상태값 등을 구체적으로 명시하여 설계와 구현 단계에서 바로 활용할 수 있도록 한다.

4. 접근성 및 포용성
- 웹 접근성 기준은 WCAG 2.1 수준을 만족해야 한다.
라벨 제공
색상 대비 확보 
키보드 내비게이션 지원
오류 메시지 설계 방식 
- 모든 사용자를 고려한 포용적 디자인 관점에서 각 화면 흐름을 검토한다.
초보 사용자 : 단계별 안내와 도움 말 제공
모바일 사용자 : 큰 터치 영역과 간단한 화면 구성

UI 상세 내용 설계하기 
1. UI 여구사항과 표준 및 지침에 따라, 사용자의 편의성을 고려한 메뉴 구조를 설계한다.
- 요구사항 확인
- UI 설계서 표지 및 개정이력 작성 
- UI 구조를 설계한다
ㄴ 요구사항 재검토, 프로토타입 재검토 
- 메뉴 구조 설계 
ㄴ 사이트 맵 (트리구조, 도형, 테이블 형태) 
ㄴ 프로세스의 정의 (도형)
- 개발 화면 설계
ㄴ UI 프로토타입과 프로세스 정의를 기반으로 개별 화면을 구체화 
ㄴ 필요한 화면의 개수를 산정
ㄴ 각 화면에 고유 ID 부여 -> 표로 작성
ㄴ 화면별 세부 내용을 설계 

2. UI 요구사항과 표준, 지침을 토대로 하위 시스템 단위의 내부, 외부 화면과 입력 폼 설계 
- 실행 부담을 줄이기 위한 UI 설계 원칙을 검토한다.
ㄴ 사용자 의도 파악 
- 행위 순서를 검토
ㄴ 사용자가 시스템 기능을 이용하기 위해 어떤 사전 행위가 필요한지 목록화 하고 조사한다.
ㄴ 각 기능의 상대적 중요도를 상,중,하로 구분하고, 구분 결과가 타당한지 비교 분석한다.
ㄴ 중요도가 높은 기능이 실행되기 위해 필요한 화면 이동이 최소로 설정되어 있는지 분석
ㄴ 전반적으로 각 기능을 수행하기 위한 이동 수가 적절한지 검토한다.
- 평가 부담을 줄이기 위한 UI 설계 원칙을 검토한다.
ㄴ 키 조작 결과를 사용자가 신속하게 인지할 수 있는지 확인
키 조작 : 키보드, 마우스, 버튼 클릭, 메뉴의 선택, 아이콘 클릭
ㄴ 각 화면 설계 결과에서 메뉴, 아이콘이 활성화 될 때, 알맞은 포인터 형태가 적용되는가 검토 
ㄴ기능이 실행되는 동안 적절한 포인터 형태가 유지되는지 점검한다.



- 검토 결과 반영하기
ㄴ 스토리 보드를 구성한다.
스토리보드 : 화면의 구성과 흐름을 시각적으로 정리한 문서
화면 ID, 화면 명, 화면 구성(검색창,검색 버튼, 카테고리)
화면 설명 : 각 UI 요소가 어떤 역할을 하는지 설명
사용자 행동 : 사용자가 화면에서 할 수 있는 행동을 적는다.
시스템 반응 : 사용자 행동 후 시스템이 어떻게 반응하는 지 적는다. 

- 상호작용 시뮬레이션을 한다
사용자가 실제로 화면을 조작하는 것 처럼 테스트하는 과정 
ㄴ 컴포넌트, 스크립트를 수정해서 정확한 상호작용을 출력한다.
상호작용을 연결 
컴포넌트        사용자 행동    출력결과
로그인 버튼   클릭                로그인 성공 시 도서 검색 화면으로 이동
검색 버튼       클릭                검색 결과 목록 표시
ㄴ 평가 진행자가 목표를 지시하고 유사 질문과 상황 통제를 통해 확인한다.
EX) 로그인 하세요, '자바'라는 책을 검색하세요, 책을 선택하세요, 도서를 대여하세요
ㄴ 사용자가 어려움을 겪는지 체크

유사 질문
메뉴 이해 확인            도서를 검색하려면 어디를 누르시겠습니까?
버튼 의미 확인            이 버튼을 누르면 어떤일 일어날 것 같습니까?
오류 대응 확인            오류 메시지를 보고 무엇을 수정해야 하는지 알 수 있습니까?
흐름 이해 확인            다음 단계로 가려면 무엇을 해야 합니까?

상황 통제 예시 
검색 결과 없음             안내 메시지를 이해하는가?
비밀번호 오류              오류 원인과 해결 방법을 아는가?
대여 불가 도서 선택     대여 불가 상태를 인식하는가?
네트워크 오류              재시도 방법을 알 수 있는가? 

사용자가 어디에서 멈췄는가?

오류 메시지를 보고 해결할 수 있었는가?
키보드만으로 조작할 수 있는가?

- 스토리보드 및 시뮬레이션의 평가 결과를 정리하여 반영한다.

메뉴명이 어렵다 -> 사용자에게 익숙한 용어로 수정

ㄴ 평가에 활용한 체크리스트는 이후 반복 검증을 위해 기록하여 관리한다.


- 사용성에 대한 전문가 및 내부 개발자가 검토 및 검증을 수행한다.
ㄴ 스타일 일관성 평가
색상 : 주요 버튼, 경고, 성공, 비활성 색상이 일관적인가?
서체 : 글꼴과 크기는 화면 마다 통일 되어 있는가?
아이콘 : 같은 기능에 같은 아이콘을 사용하는가?
배치 : 로고, 메뉴, 버튼 위치가 일정한가?
여백 : 요소 간 간격이 일정한가?

- 컴포넌트 활용도 평가
기본 버튼 : 로그인, 검색, 대여, 반납
검색창 : 공지사항, 도서 목록
목록 테이블 : 공지사항 목록, 도서 목록, 대여 목록 
팝업 : 대여 완료, 반납 완료, 삭제 확인 
상태 배지 : 대여 가능, 대여 중, 반납 완료

컴포넌트의 활용도가 낮은 경우
화면마다 버튼의 크기와 모양이 다를 수 있다.
같은 검색창인데 위치와 동작이 다를 수 있다.
팝업 디자인이 화면마다 다르고 
상태 표시 방식이 통일되어 있지 않다.

- 서비스 패턴 준수 여부를 평가한다.
핵심 기능 흐름이 필수, 권장, 우수 수준에 맞는지 평가한다.

검색 패턴 평가 예시 
필수 검색어 입력, 검색 버튼, 결과 목록 제공
권장 최근 검색어, 자동 완성, 검색 결과 개수 표시
우수 맞춤 추천 검색어, 오타 보정, 인기 검색어 제공 


- 접근성 충족 여부를 확인한다.
- 다양한 화면/기기에서 동일하게 구동되는지 확인한다.
PC : 메뉴, 목록, 표가 정상적으로 보이는가?
태블릿 : 버튼 터치가 쉬운가?
스마트폰 : 화면이 잘리지 않고 세로 배치가 적절한가
작은 화면 : 검색창과 버튼이 겹치지 않는가
다른 브라우저 : 기능이 동일하게 동작하는가?
