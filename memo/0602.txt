2. 물리데이터 저장소 설계하기
반정규화
- 조회 성능을 높이기 위해 데이터를 중복하거나 테이블 구조를 바꾸는 설계 방법
반정규화 유형
- 중복테이블 추가(집계, 진행상태,특정 컬럼을 포함)
- 테이블 조합(같이 사용되는 두 개 이상의 테이블을 하나로 합침)
- 테이블 분할(수평, 수직분할)
- 테이블 제거

물리데이터 모델링 하기

컬럼의 타입과 길이를 정의(프로젝트 할 때 정한 DBMS를 따라가면 된다.)
ORACLE
- 문자열 : CHAR, VARCHAR2(길이)
- 숫자 : NUMBER(자리수,소수점)
- 날짜 : DATE

MYSQL
- 문자열 : VARCHAR(길이)
- 숫자 : INT, DECIMAL
- 날짜 : DATE, DATETIME

3. 물리 데이터 저장소 구성
- 설계에 따라 데이터 저장소에 실제 데이터가 저장될 물리적 공간 구성

테이블 제약조건
- 테이블간 관계가 설정되었을 때 데이터가 잘못 저장된거나, 관계가 깨지는 것을 막기 위한 규칙

기본키 테이블의 데이터가 삭제 될 때
- CASCADE : 기본키의 데이터가 삭제되면 참조하는 데이터도 삭제
- RESTRICT : 데이터가 참조되고 있다면 기본키의 데이터는 삭제할 수 없다
- NULLIFY : 기본키의 데이터가 삭제되면 참조키 데이터를 NULL로 바꿈

사용법
CONSTRAINT 외래키명 FOREIGN KEY (컬럼명)
REFERENCES 테이블명(컬럼명)
ON DELETE 제약조건

기본키 테이블의 데이터가 수정 될 때
- CASCADE : 기본키의 데이터가 삭제되면 참조하는 데이터도 수정
- RESTRICT : 데이터가 참조되고 있다면 기본키의 데이터는 수정할 수 없다
- NULLIFY : 기본키의 데이터가 수정되면 참조키 데이터를 NULL로 바꿈

사용법
CONSTRAINT 외래키명 FOREIGN KEY (컬럼명)
REFERENCES 테이블명(컬럼명)
ON UPDATE 제약조건

인덱스 설계
적용되는 컬럼의 선정
분포도가 10 ~ 15%인 컬럼에 사용하는것이 일반적임
분포도 = (1/컬럼값의 종류) X 100

분포도가 범위 이상이라도 부분 처리를 목적으로 하는 경우
조회 및 출력 조건으로 사용되는 컬럼인 경우
기본키(PK)와 유일키(UNIQUE)는 인덱스가 자동으로 생성된다

분포도가 좋은 컬럼은 단독 인덱스로 생성한다.
자주 조합되어 사용되는 컬럼은 결합 인덱스로 사용

수정이 빈번하지 않은 컬럼으로 선정

뷰
뷰 속성
REPLACE ( CREATE OR REPLACE 뷰이름)
- 뷰가 이미 만들어져 있다면 기존 뷰를 새로운 내용으로 다시 생성하는 속성

FORCE
- 기본테이블이 없어도 뷰를 생성하는 속성
- 대신 뷰를 사용하려고 하면 에러 발생

NOFORCE
- 기본테이블이 없으면 뷰를 만들지 마라

WITCH CEHCK OPTION
- 뷰의 조건을 만족하는 행만 INSERT 또는 UPDATE 할 수 있다.

WITH READ ONLY
- 뷰를 조회 전용으로 만드는 속성

뷰 설계시 고려사항
- 복잡한 뷰는 성능이 느려질 수 있다.
- 뷰 안의 SELECT 조건도 인덱스를 고려해야 한다.

클러스터 설계
- 관련된 데이터를 물리적으로 가까운 위치에 저장하여 조회 성능을 높이는 기법

인덱스 : 목차
ㄴ 특정 행의 검색이 빠르다
클러스터 : 물리적으로 모아둠
ㄴ 범위 조회, 반복 조인 조회가 빠르다.

클러스터 만드는법
1. 클러스터 생성
CREATE CLUSTER 클러스터이름(
	컬럼명 타입;
)

2. 클러스터 인덱스 생성
CREATE INDEX 인덱스명
ON CLUSTER 클러스터명;

3. 클러스터를 사용하는 테이블 만들기
CREATE TABLE 테이블명(
	...
)
CLUSTER 클러스터이름(컬럼)

MYSQL은 오라클처럼 직접 만들지는 않는다.
PK를 기준으로 자동으로 클러스터형 인덱스만들어진다.

적용기준
- 분포도가 넓을 수록(수치가 낮은것) 유리한 기법
- 클러스터는 여러 블록이상의 테이블에 적용한다.(작은 테이블에는 효과가 크지 않다.)
- 대량의 범위를 자주 조회할 때 적용
- 여러 테이블이 빈번히 조회될 때

주의사항
- 검색은 빨라지지만 입력,수정,삭제는 느려질 수 있다.

파티션
- 하나의 큰 테이블을 내부적으로 여러 조각으로 나눠 저장하는 기술

파티션 종류
범위 분할
- 사용자가 데이터를 구간별로 나눈다.
- 데이터가 각 파티션별로 골고루 들어가있지 않을 수도 있다.
- 조회 범위를 줄일 수 있다.
- 오래된 데이터를 관리하기 쉽다.
- 주문,로그, 게시글, 이력데이터

해시 분할
- DB가 계산해서 각 파티션에 골고루 분배
- 해시값 % 파티션 개수 -> 어떤 파티션에 넣을지 결정
- 회원, 사용자,고객 데이터 -> ID기반의 조회가 많은 시스템
조합 분할
- 파티션 기준을 2개 이상 조합해서 데이터를 나누는 방식
- 먼저 큰 기준으로 한 번 나누고 그 안에서 다시 작은 기준으로 한 번 더 나누는 방식
- 1차 파티션만으로 부족할 때
- 날짜 조건과 고객 조건을 함게 자주 사용할 때
- 특정 연도나 월에 데이터가 몰리는 문제를 줄일 수 있다.

파티셔닝 순서
- 파티션 종류 결정(범위,해쉬,조합)
- 파티션 키 선정(데이터를 나눌 기준이 되는 컬럼)
- 파티션 수 선정

디스크 구성 설계
- 데이터를 어디에, 어떻게 저장할 것인지 계획하는 작업
- 정확한 용량을 산정하여 디스크 사용의 효율을 높인다.
- 업무량이 집중되어있는 디스크를 분리하여 설계함으로써 집중화된 디스크에 대한 입출력 부하를 분산한다.
- 입출력 경합을 최소화하여 데이터의 접근 성능을 향상시킨다.

물리 데이터 모델 구성하기
테이블 이름 정하기, 컬럼 이름 길이 정하기, 기본키 외래키 정하기

요구사항이 잘 반영됐는지 확인
DBMS 선택(ORACLE, MYSQL, MONGODB...)을 하고 학습

수행순서
다양한 오브젝트를 설계(테이블,제약조건,인덱스,뷰,클러스터,파티션,테이블스페이스)
ㄴ 제약조건 설계 : 데이터 무결성 보장
ㄴ 인덱스 설계 : 검색 속도 향상
ㄴ 뷰 설계 : 재사용성과 보안
ㄴ 클러스터 설계 : 물리적 저장 최적화
ㄴ 파티션 설계 : 대용량 데이터 관리
ㄴ 디스크 구성 설계 : I/O 성능 최적화

IF문
DECLARE
	v_score NUMBER := 85;
BEGIN
	IF v_score >= 60 THEN
		DBMS_OUTPUT.PUT_LINE('합격');
	ELSE
		DBMS_OUTPUT.PUT_LINE('불합격');
	END IF;
END;

LOOP : 같은 작업을 반복할 때 사용
DECLARE
	v_score NUMBER := 85;
BEGIN
	LOOP
		DBMS_OUTPUT.PUT_LINE(v_score);
		v_score := v_score + 1;
		EXIT WHEN v_score > 90
	END LOOP;
END;

ERROR 처리
EXCEPTION을 이용하여 오류를 처리할 수 있다.

DECLARE
    v_name member.name%TYPE; -> member테이블의 name컬럼의 타입과 길이를 쓰겠다.
BEGIN
    SELECT name
    INTO v_name
    FROM member
    WHERE member_id = 999;

   DBMS_OUTPUT.PUT_LINE(v_name);

   EXCEPTION
   WHEN NO_DATA_FOUND THEN
   DBMS_OUTPUT.PUT_LINE('조회된 회원이 없다')
END;
/

Stored Procedure
- function은 값을 반환하는데 초점이 있다면, procedure는 작업 수행에 초점이 있다.

in 매개변수 -> 외부에서 값을 받아오는 매개변수
out  매개변수 -> 외부로 값을 내보내는 매개변수
inout 매개변수 -> 값을 받아와서 처리한 다음에 내보내는 매개변수

create or procedure 프로시저명(
	매개변수
)
IS
	지역변수
BEGIN
	트랜잭션
END;

바인드 변수(전역변수)
VARIABLE 변수명 타입;

OUT매개변수가 있는 경우

INOUT매개변수

EXEC 프로시저명(:변수명);
