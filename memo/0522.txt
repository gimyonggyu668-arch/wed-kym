논리 데이터 모델링 
- 데이터베이스 개발 과정의 첫 단계
- 업무 분석을 통한 자료의 흐름을 분석하여 현재 사용중인 양식, 문서, 장표를 중심으로 자료항목을 추출하여 엔티티와 속성들의 관계를 구조적으로 정의하는 단계 

특징 
- 논리 데이터 모델링을 진행할 때 요구사항을 충분히 수집하지 않으면 요구사항 변경에 따른 많은 비용이 발생한다. 
- 이해당사자들과 의사소통의 보조 자료로서 E-R 모델을 활용한다.
- 논리적 모델은 H/W나 S/W에 독립적이다.


함수적 종속성
- 어떤 속성의 값이 정해지면 다른 속성의 값도 하나로 정해지는 관계 

A -> B
A가 B를 결정한다.

학번                    이름              학과
2026001           김철수           컴퓨터공학
2026002           이영희           전자공학
2026003           박민수           경영학


학번은 학생마다 유일하다.
학번 -> 이름 : 학번을 알면 이름을 알 수 있음
학번 -> 학과 : 학번을 알면 학과를 알수 있음

학번          과목번호           성적
1001         J01                   A
1001         P01                   B


기본키 : (학번, 과목번호)로 이루어진 복합키 
학번만으로는 성적을 모름
과목번호만으로도 모름 
둘 다 있어야 성적을 알 수 있다.

부분 함수 종속 (Partial Dependency)
기본키의 일부만으로 결정되는 경우

학번           과목번호         학생이름
1001         J01                  김철수

기본키 : (학번 + 과목번호)
학번 -> 학생 이름


3. 이행함수종속(Transitive Dependency)
중간 속성을 거쳐 결정되는 경우

학번           학과번호        학과명
1001          D01               컴퓨터공학

학번 -> 학과번호
학과번호 -> 학과명
학번 -> 학과명 

정규화와 연결하면
1단계 : 함수적 종속성 찾기
2단계 : 부분 함수 종속을 제거
3단계 : 이행 함수 종속을 제거
4단계 : 다치종속을 제거 


다치종속
- 하나의 속성이 여러 개의 값을 가지는 데, 그 값들이 다른 속성과 서로 독립적인 경우 

A는 여러 개의 B를 가진다.
학생            취미               자격증
김철수        축구               정보처리기사 
김철수        축구               SQLD
김철수        게임               정보처리기사 
김철수        게임               SQLD

취미와 자격증은 서로 관련이 없다
그런데 하나의 테이블에 넣으면 축구 + 정보처리기사, 축구 + SQLD, 게임 + 정보처리기사, 게임 + SQLD의 조합이 생기게 된다. 

학생 - 취미
학생 - 자격증
ㄴ 이렇게 테이블을 나눠야 한다.


조인종속 (JOIN Dependency)
- 하나의 테이블을 여러개로 분리했다가 다시 조인해야만 원래 의미가 유지되는 관계 

교수             과목
홍길동         java
홍길동         DB


교수              강의실
홍길동          101호
홍길동          102호


과목             강의실
java             101호
DB               102호


select * 
from 교수 - 과목
join 교수 - 강의실
join 과목 - 강의실

논리 데이터 모델 검증하기 
1. 업무 분석가, 데이터베이스 엔지니어가 작성한 논리 데이터 저장소 설계 내역에서 정의된 데이터의 유형을 확인하고 식별한다.

구축 시스템의 업무 관련 데이터 유형을 식별한다.
- 업무 처리를 위한 입력 데이터와 출력데이터가 식별되었는지 확인
- 각 프로세스별 입력 데이터 및 출력 데이터 목록 데이터 목록을 도출하고, 설계서에 정의된 항목과 일치하는지 비교
 
누락된 항목이 있는 경우, 업무 흐름도 또는 유스케이스를 참조하여 보완한다


관련 업무와의 연계 데이터가 식별되었는지 확인한다.
- 타 부서 혹은 외부 시스템과 주고 받는 데이터 목록을 추출한다.
- 인터페이스 명세서, api정의서 등을 참고하여 연계되는 데이터 항목을 정리하고
내부 시스템과 외부 연계 기관의 송수신 항목을 구분한다.
- 연계 대상 시스템, 인터페이스 형태 (REST, DB LINK등)

2. 현재 운영 중인 시스템의 데이터 현황 분석 수행하기
- 운영중인 DBMS의 테이블 구성, 파티셔닝, 이중화 구성 등을 체크한다.
- 백업 정책이 문서화되어 있는지 확인한다.
- 데이터 속성, 공통 코드 정의 현황을 분석한다.
- 공통 코드 테이블이 존재하는지, 일관되게 참조되고 있는지 점검한다.
- 외부 연계 데이터 속성을 확인한다.
- 데이터 정합성, 필수 여부, 오류 처리 로직 등이 정의되어 있는지 확인한다.

3. 사용자 요구사항이 반영되었는지 분석한다.
- 저장 데이터 볼륨, 분산 구조, 제약조건에 관한 요구사항을 확인한다.
- 타 시스템과의 연계 요구사항을 확인한다.
- 데이터에 대한 백업 및 복구 정책 요구사항을 확인한다.
- 초기 데이터 구축 요구사항을 확인한다.
- 기존 데이터 전환 요구사항을 확인한다.
- 무결성 및 데이터 보안성 요구사항을 확인한다.

논리 데이터 저장소 설계 내역에서 데이터의 논리적 단위와 데이터 간의 관계를 확인한다.
1. 데이터 흐름 정의 상태 점검하기
- 각 프로세스 흐름도를 분석하여 어떤 엔티티를 참조하거나 변경하는지 확인한다.
- 입출력 데이터가 설계서 상에서 정확히 반영되어 있는지, 누락되거나 과도하게
정의된 항목이 없는지 점검한다.
2. 프로세스간 공유되었거나 연계된 데이터의 정의 여부를 확인한다.
- 하나의 엔티티가 여러 프로세스에 의해 공유되는 경우, CRUD 행위별 주체가 명확히 구분되었는지 확인한다.

설계 기준 정의 상태를 점검한다.
- 데이터베이스 설계 표준 지침 확인하기
테이블/컬럼명 명명 규칙, 기본키/외래키 정의 원칙, 속성 유형 정의 규칙 등이 포함되었는지 확인한다.

- 데이터 모델 설계 지침을 확인한다.
정규화 / 반정규화 원칙, 엔티티 분리 기준, 속성 설계 기준이 정리되어 있는지 확인한다.



검토관점 
필요한 자원
- 시스템을 구현하고 운영하기 위해 얼마나 많은 컴퓨터 자원과 개발 자원이 필요한지 확인
상호 운용성
- 우리 시스템이 다른 시스템과 정보를 주고 받을 수 있는지 의미
시장 성숙도 및 트렌드 부합성 검토
- 사용하는 기술이 충분히 안정적이고 많이 사용되고 있는지 확인하는 것이다.
- 트렌드 부합성은 현재 개발 환경이나 산업 흐름에 맞는 기술인지 확인하는 것
- 기술을 잘못 선택하면 개발 중 문제가 생길 수 있다.  
기술적 위험 분석
- 개발 과정에서 문제가될 수 있는 위험 요소를 미리 찾는 것이다. 

타당성 분석 수행
순서
1. 분석 모델까지 요구사항 추적표를 작성하고, 타당성 검토 의견 속성을 추가한다.
2. 작성된 요구사항 추적표에 타당성 검토 의견을 작성한다.
- 성능 및 용량 
- 시스템 간 상호 운용성
- 시장 성숙도 및 트렌드 부합성
- 기술적 위험성 분석을을 참조하여 검토 의견을 작성한다.

각 모델별로 유의해야 할 점
유스케이스 모델 타당성 검토
- 관련 액터들이 너무 많지는 않은지 
- 이벤트 흐름과 예외 흐름이 지나치게 복잡한지는 않은지

개념 수준 분석 클래스 모델 타당성 검토
- 너무 많은 클래스 모델들이 도출되지 않았는지 

개념 수준 분석 클래스 모델 
- 요구사항을 읽고 시스템에서 관리해야 할 중요한 사람, 사물, 정보, 업무 개념을 클래스로 뽑은 것
ex) 도서 대여 시스템

분석 클래스 모델 타당성 검토
- 요구사항을 만족시키기 위한 경계 / 제어 / 엔티티 클래스들이 너무 많지는 않은지 
- 관계가 너무 복잡하지는 않은지
- 관리해야할 속성들이 지나치게 많은지 검토 

분석 클래스 모델 
- 요구사항을 만족시키기 위해 필요한 클래스들을 역할에 따라 정리한 모델
경계 클래스 : 사용자나 외부 시스템과 만나는 클래스 
제어 클래스 : 기능 흐름과 처리 순서를 담당하는 클래스 
엔티티 클래스 : 실제 업무 데이터를 저장하고 관리하는 클래스 


타당성 분석 결과를 이해관계자가 검증한다.
1. 타당성 분석 결과를 관련 이해관계자에게 배포하여 사전 검토를 요청한다. 
2. 관련 이해관계자가 모여 분석모델 타당성 분석 결과를 검증한다.
3. 타당성 분석 결과에 이견이 있는 경우 프로젝트 관리자의 중재하에 합의를 도출한다. 

관련 이해관계자 검증을 거친 타당성 분석 결과를 확인하고 배포 및 공유한다. 
1. 관련 이해관계자 검증을 거친 타당성 분석 결과를 의사 결정자가 확인한다.
2. 의사 결정자가 확인한 타당성 분석 결과를 이해관계자에게 배포하여 공유한다.
